3.2. The command file

If you simply issue the command os9disasm <programfile> on a program file, the resultant code generated should assemble back into an identical module or program file as that from which it was disassembled. However, in this form, the source might not be very intelligible in places. Without some direction, OS9Disasm blindly disassembles all byte sequences as machine code. Most programs have non-code areas - string data, lookup tables, and other data - which, if parsed as machine code, produces nothing but gibberish from an assembly perspective. Often the data being disassembled in this way will not work out to end at the actual end of the data area, and genuine machine code which follows will be obliterated, also. This is where the command file comes into play.

3.2.1. Data bounds

Data areas can be specified in different ways. These areas are, as previously mentioned, areas of data storage which are not assembler code, but rather are constant values that the program might use. These can be string constants, lookup tables, or simply numeric constants. These blocks of data code can be specified in one of five ways

  1. ASCII data will generate FCB, FCC, or FCS (if appliable) statements with strings automatically delimited, and ASCII control characters delimited by their standard names.

  2. Word data areas will produce FDB's of hex data double bytes.

  3. Byte data areas will produce FCB's of hex data byte.

  4. Long label data areas will produce FDB's of label references to the program. Many programs have data tables in them which are used for indirect addressing or for performing multi-direction jumps. Such tables are best disassembled as a seried of labels.

  5. Short label data areas are the 8-bit equivalent of Long labels. These will produce FCB's of labels.

The generation of labels in operands can be completely controlled. Whenever a label might conceivably be generated in an instruction, you can specify whether a data value or a label should be used. Further, it is possible to specify labels in several label classes. With this ability, the address $10 in the direct page instruction LDA <$10 can be given a different label than that given the address $10 in the extended instruction LDB >$0010.

All calls to the OS/9 operating system via the 'OS9' opcode can be automatically disassembled using the standard system names. This automatic generation of 'OS9' calls can be turned off if desired.
