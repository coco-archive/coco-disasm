Installation

GIT Clone

This package can be cloned from its Github repository. The address is https://github.com/debmint/coco-disasm The commandline command is 

# git clone https://github.com/debmint/coco-disasm.git

This will clone the project into the directory coco-disasm . You can specify a different directory name by appending this name to the end of the command to clone the project.

Tarfile extraction

If the package was downloaded as a tarfile, it should have the extension of tar.bz2 or tar.gz. For either case, cd into the directory in which you will install the directory tree and issue the command

for the tar.bz2 extension 

# tar -xjf <path>/.tar.bz2

OR, for the tar.gz extension.

# tar -xzf <path>/os9disasm.tar.gz

In either case, a directory named cocodisx.x will be created in the current location, and all files will be created under that directory.

Building and Installing the Package

The packages now comes configured with autotools. This means that the configure process should be able to detect the operating system under which the process is running and set some compile parameters to suit your situation. This should make it easier for the programs to be compiled under differing systems.

Running the configure script

The first thing you need to do after cd'ing into the top level of the distributions directory is to issue the command ./configure. This will inspect the system under which it is running and create various files, and especially the Makefiles that will handle the actual building of the system.

Don't mess with the configuration files!

Unless you know what you're doing and see that something in the configure setup needs to be fixed, please heed the following.

There are a few files that you should never touch unless you have the autotools package. If the timestamp of any of these are not as they should be, a complete reconfigure may be required. If you do not have the autotools utilities, you cannot do this, and even if you do, if it's version is not at least that of the one which originally built the configure scripts, then a complete reconfigure might be required.

The files that should not be touched are as follows:

  • Makefile.am
  • Makefile.in
  • configure.ac
  • configure.in
  • aclocal.m4

If you do find that the system is trying to do a reconfigure and you are stuck, you can always remove your install of the package and re-extract it from the original tarfile or do 

# git checkout master

Options passed to configure

You can review the options accepted by configure by issuing the command ./configure --help. Most are the standard options that all configure scripts accept, but there are a few specific to this build.

--with-g09dis

G09Dis by default is not compiled with OS9Disasm. The reason for this is that G09Dis requires the GTK package to build and run. In order to compile G09Dis, pkg-config must be installed on the system. Since some systems might not have this, the configure process does not include g09dis in the compilation. If this option is not included, the Makefile for G09Dis will be made, but will not be correct, and the g09dis directory will not be processed when the make command is issued.

--with-windis

WinDis, as the above, by default, is not compiled with OS9Disasm. If building on a Linux system, the necessary libraries are likely not installed and would produce errors if attempting to build.

If this option is not included, the Makefile for WinDis will be made, but will not be correct, and the windis directory will not be processed when the make command is issued.

Building the program(s)

After running configure you are now ready to build your program(s). You build the programs by issuing the command make. This will build your program, but won't install it.

Note

You are not required to build from the top-level directory. The system is designed such that you can cd into any subdirectory and run make on just that directory. It will, of course, recurse into any subdirectories below that level, but won't bother anything in a sibling or higher directory outside that particular level.

This holds true for any of the make commands. The command will apply over only that directory and any directories below it. No higher directory or any outside this directory will be involved.

In addition, you can also issue the command make <component> to build a single aspect of the suite.

Packages needed to build the programs

On a Linux system, most packages needed to build OS9Disasm will likely already be installed. If not, with most distros, they will be readily available. If any needed packages are not installed, it should be seen when configure is run.

Packages needed by G09dis

To build and run g09dis, several packages are needed. Even if running a precompiled package, several GTK runtime packages will be needed.

In addition to the runtime packages, the corresponding -dev packages will be required to build the system. Some of the packages needed are:

gtk+
glib
pango (possibly)

In addition, to compile, pkgconfig is essential.

Note that these package names will be accompanied with a version number, also. GTK and glib should be version 2 or greater. Also remember that each of the above runtime packages (except for pkgconfig) will have a "-dev" counterpart that will be needed for compiling ( but not for running).

Building and/or running on Windows

These packages can be built and run on Windows systems. The first requirement, and this applies to both OS9Disasm and GTK, is the linux interface by which you can compile the programs. The easiest setup for building under Windows is to have the msys2 installed. All the packages needed to build are available and it should be pretty straightforward to do a successful build.

it should be possible to build under CygWin, but the configuration files for this package would probably need to be altered.

The GTK interface is available for Windows, and OS9Disasm, of course, can be run from the command line without GTK if you do not wish to bother with GTK.

The GTK interface is available by several means. There are at least two executable install systems available (cannot recall the sources at the present), or the various modules can be installed piece by piece.

In my opinion, the piece-by-piece installation is better in that, if one particular package is upgraded, you can simply overwrite the existing package and not have to do a complete reinstall. Tor Lillqvist maintains a pretty good and web page for this, accessible through either the gtk.org or the gimp.org website, following the links for Windows packages. These are zip files, and you can begin by installing only the ones you think you will need, and, if you need something else, go back and install it in addition.

Installing the program(s)

To install the program, you can issue the command make install (at any directory level you choose). This can be done whether the program has already been built or not. If it hasn't been built, issuing this command will cause anything that would need to be done before installing to be done before attempting to do the install. By default, this will install the OS9Disasm binary in the directory /usr/local/bin/.

Note that on most systems, to install to a standard location, such as /usr/local/bin/, you need superuser priviledges.

Installing in an alternate location

If you wish to install it elsewhere, then you can either simply copy the binary files to whatever destination you want after running the command make, or, if you wish to install the program(s) at a different location using make install, then you can include the option --prefix=<alternate-location> when you run configure.

Install the binaries where the system can find them

It may be very advisable to carefully contemplate the location where you will install the binaries. This is especially true for OS9Disasm if you are planning on calling it from within g09dis. If you disassemble a binary from the menu in g09dis, a shell will be issued which will call up OS9Disasm to do the work. The call will be of the form 

"os9disasm program <etc>"

It would be a bit difficult - and harsh - to hardcode a specific location into the program whereby everyone would be required to install OS9Disasm at some specific location. There are just too many differences between the various systems and people have preferences as to their install locations. Therefore the call is called up in somthing of an open manner, and it's left up to the operating system to find OS9Disasm. The following gives some insight in how this can be assured.

Linux and Unix-type systems.  To find the directories searched in locating an executable program, from the shell prompt, issue the command 

echo $PATH

This will give a list of directories, separated by a colon, searched when looking for a command. They are searched in the order in which they appear.

Note

For security reasons, be aware that most Linux/Unix systems don't include the current working directory in the search path.

Windows Command prompts. If you're running from a POSIX shell, (msys, cygwin, or sh/bash/etc from an MS window, you can view the search PATH with the command as above, 

echo $PATH

or, if from the MS command, 

echo %PATH%

In msys/cygwin, you can add more search directories by editing one of the configure files. Please see the documentation for the system for details. To alter the windows PATH, you will need to edit the autoexec.bat file. To do this, you can either add your new directory to an existing PATH statement or add a line such as 

set PATH=%PATH%;your-new-directory

Note that on most systems in Windows, the current working directory is normally included in the search path. Therefore, you should be able to call OS9Disasm successfully if OS9Disasm is in the directory you are in when you call OS9Disasm. In other words, if the command 

ls os9disasm

or 

dir os9disasm

if you are at a DOS prompt succeeds, then you should be OK if you call g09dis as long as you don't cd somewhere else before doing so.

Note

You can alter the PATH variable at any time by by issuing the above command in an MS window, but this change will be lost when the window is shut down and won't be in effect when starting a new window, or won't be available to the global Windows system.

Running g09dis from the Explorer Window. I'm not entirely familiar with how this will work. If you do this, probably the safest route would be to make certain that OS9Disasm is in a directory listed in PATH.

The Documentation files

Due to all the differences in systems, it has been decided at this time to not do an install of the documentation with a make install command from a higher directory. The html files are already built and are located in the doc/html directory. These can be manually copied to any location you desire. In addition, the xml source is available in the doc directory, and fresh html documentation can be created by issuing the command make in the doc directory itself.

Building your own documentation. Since the source is available in the doc directory, you have the option of building the documentation in any format you desire. Just run any docbook compiler on os9disasm.xml and create any output you wish. The file os9disasm.xml will include all the rest of the files.

Installing the Standard Label Files

In the top level directory, there is a directory named defs. In this directory, you will find several files ending in "lbl". These are the standard label files.

Copy all these files into whatever directory you wish your DEFS files to reside. The default is /dd/defs. wish your "defs" directory to reside. /dd/defs is the default location, but you can specify any path with the -d command-line option.

Getting /dd on a unix-style system.  There are a couple of ways to get /dd to exist on a unix-style system. The first is simply to create a directory named dd in the root directory and place all the files there. However, I preferred to have my coco files in my $HOME directory. This way, I did not have to worry about getting root permissions to modify files in this directory, so I created a coco directory there and created a symbolic link to that directory in the root directory.

On Windows, if you operate under MinGW, the simplest way is to create a directory /dd in the root directory. Note that symbolic links are not available under Windows.

In addition to these options, there is an option in OS9Disasm -d which allows you to specify an alternate directory where the defs files are found. This can be embedded in your command file for each disassembly you are doing.
