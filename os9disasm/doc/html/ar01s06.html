G09Dis - A graphical front-end to OS9Disasm

OS9Disasm is a command-line application. It is operated from the command line and uses the various files to perform its work. This is all well and good, but it can become tedious trying to remember all the Addressing modes and such when attempting to add entries to the command file, especially.

This is where a helper application can be beneficial. This is just what G09Dis is. With it, you can add command lines to the command file or the label file, depending upon the content of the line upon which you are working.

Installing and Compiling G09Dis

The installation and compilation is in principle the same as the general installation, so you are encouraged to read the section called “Installation” for instruction on this.

Components of the G09Dis Window

The Listing Window

The Listing Window appears on the left side of the displayed window. This is a display of the formatted output produced by OS9Disasm, and is equivalent to the output of an assembler when most are normally given the -l option.

The Listing Window is where you will need to reference your work. It is from a reference to a line in the Listing where you add the commands or labels. This listing is the Listing output from OS9Disasm.

Do not use a regular Listing!

Be sure you do not try to load a regular listing. These WILL NOT LOAD. You MUST use a specially formatted Listing! See below for details.

Obtaining and Loading a compatible Listing for G09Dis

As stated above, a normal listing will not work. If you attempt to use one of these, nothing will happen, nothing will be loaded. You must use a specially formatted listing using tabs to separate the fields rather than spaces. This can be obtained in one of two ways.

  1. Generate a separate file externally with OS9Disasm using the -g option, redirecting the output to a file and then loading from File->Load Listing in the main menu, or
  2. Check the Disassemble to GUI option in Options->Disassembler Options and then Tools->Disassemble in the Main Menu. This will disassemble the program and load the Listing directly into the GUI window. No file will be generated in this case.

The Command File Window

The command window is displayed in the top half of the right-hand side of the display. This displays the command file as used by OS9Disasm. It is either loaded from an file or created here.

This section is editable, either directly within the window, or commands can be added from popup windows depending upon the context of the line in the Listing Window. See the section called “The Define Bounds Window” and the section called “The Set Addressing Mode Window”.

The Label File Window

The Label file window is in the lower half of the right side of the display. This represents a label file for the program you're working on. Although OS9Disasm supports up to 16 label files, G09Dis supports displaying only one label file at a time. This may change in the future.

This window is not editable directly, but lines can be added by accessing the "Rename Label" popup.

Running G09Dis

Operation of G09Dis is relatively straightforward. A few notes are in order.

The File Menu

The New menu item

The New menu item clears the specified window and the associated buffer of data. When the New menu item is clicked, it causes a popup submenu to appear. In this menu, you can choose which window you wish to clear. Each window is handled individually.

The Load menu item

Clicking on this menu item produces a submenu window similar to the New to appear. This will bring up a File chooser window which will allow you to choose a file to load into the corresponding buffer and it will appear in the corresponding window. If a buffer for that window is already loaded, it will be cleared, and the data in the buffer and the window will be replaced.

No attempt is made to determine the file type of your choice. If an incorrect file is chosen, in most cases, the result is that the file simply will no load.

The Save menu item

This item allows you to save either the Command File or the Label File buffer to a file. If the buffer began as a file loaded from an already-existing file, or the buffer has been previously saved to a file, the buffer is saved without any prompt. If the buffer was created from scratch, and no file for that buffer has yet been accessed, you will be prompted for a filename and path to which to save, in the same manner as The Save As... menu item.

The Save As... menu item

This will allow you to specify a filename (and path) to which you will save the Command File or the Label File. A File Chooser window will appear allowing you to specify your choice. After saving from this menu, you can make further saves to this file by simply clicking the Save menu choice.

You can choose this option whether this buffer already has a filename associated with it or not. If you choose this option for a buffer that already had a filename associated with it, this new filename becomes the default name.

The View Menu

The Fonts menu item

Clicking on this item allows you to change the font and its pointsize, and the color of the text or background for any of the three display windows. Please refer to the section called “Visual Preferences” for details on this option.

The Tools Menu

Modify Amode List

This allows you to modify (temporarily for the duration of the session) the list of Addressing modes recognized by G09Dis. Definition modifications added through this menu last only for this session, and are lost when the G09Dis is closed out. To make any changes permanent for future sessions, please seethe section called “Embedding Label Classes in Files”.

Addressing modes can be added either one at a time through the entry line or they can be added from a file containing addressing mode specifications.

If the amodes are loaded from a file, a file is selected, and then it is read, and only lines beginning with the key "*\" are accepted. (see the section called “Embedding Label Classes in Files”. In this way, a label file can be read and only the amode specifications will be loaded, with all other lines ignored.

Disassemble to GUI

This will cause OS9Disasm to run, disassembling the currently specified program and sending output to the Listing display. This option is disabled unless a program has been specified by either the Options menu or by loading from a saved options file.

Disassemble to File

This option disassembles the program sending the formatted listing output to a file. This file is a standard space-separated listing and not one formatted for G09Dis (see the section called “Obtaining and Loading a compatible Listing for G09Dis”).

The Options Menu

The Options menu allows you manipulate options both for the disassembler and for preferences in the operation of G09Dis.

Elements of the Options Menu

Disassembler Options

This brings up an options window allowing you to set various options for the disassembly of the program. Most of these are options passed to the disassembler when disassembling the program. When Tooltips are turned on, a short description for most is displayed when you hover the mouse over the title momentarily.

The window is broken down into two pages. The page titled Files pertains to files OS9Disasm will use. The other page, titled Appearance deals primarily with the appearance of the formatted listing that OS9Disasm will output.

Note that to disassemble a program, an entry must be present in "Program to Disassemble". This is the only item that is definitely required. Please refer to the documentation for OS9Disasm for a more in-depth explanation of the various options.

Most of these options (with the exception of "Program to Disassemble" and the name of the command file can be specified within the command file itself, if one is used.

Load Options from File, Save Options to file

This is a convenience feature. If you are working on a larger project, you often won't complete it in one session. Rather than inputting all the data each time, you can, after setting the parameters, click "Save Options to File" and save the settings. Then, when you come back to work on the project further, simply reload the setting with "Load Options from File".

Note that any of the options can still be changed during the session. Of course if you want these changes to be those you wish to make permanent, you will need to resave them.

Of course, any of these options that apply to disassembly particulars can be embedded in the command file itself. However, the primary benefit of this feature is that this will save and load the name of the Program file to disassemble and also will load the command file and the label file into the buffer for further editing.

Show Tooltips

This is a toggle option. If it is checked, many of the items appearing in the windows will show some tips if you hover the mouse over the item for a bit. Not every item has this feature, but many do have some helpful tips.

The Listing popup menu

Right-clicking in the Listing window will bring up the following popup menu:

The List Menu

Search Listing

Search Listing searches through the listing display for a string of text. The search will search for this text in a specified column. The choices for the column are Address, Label, or Opcode. Please refer to the section called “Performing Searches” for more details on searching.

Open Listing

Open Listing loads a specially formatted listing from a file.

What we are now interested in are the editing functions.

Note

In the following, the window decoration will be different, depending upon the window manager or windowing system under which you are running.

The Rename Label Menu Option

If we click on "Rename Label", we get a window similar to the following.

The Rename Label window

Here we see three input boxes. The Label Name box allows one to give the Label a meaningful name.

The Label Address box is the Address (in hexadecimal). Note that this address MUST be given in hexadecimal. The Addressing Mode is the Addressing mode under which this address will be given this name.

If you are changing a Label Name for a line under which you clicked, the Address should not be changed and in most cases, the Addressing Mode should be correct. However, you can set up the label for any address and any mode by clicking on any line and altering the address and/or mode. A new label can also be added by right clicking in the Label File window and clicking on "Insert After".

The Define Bounds Window

Next we have Define Bounds. With this menu option, we get a window similar to:

The Define Bounds Window

In this window, we define a boundary range. In the Boundary type, we have a pulldown list where we can specify the type of data this range represents. Please refer to the section called “Data Area Boundary Specification Command Line” for more details on data boundaries.

Under this is the "Addressing Mode". This is not applicable to all Boundary types. Again, refer to the section called “Data Area Boundary Specification Command Line” for more information on this. If the Boundary type does not use an Amode, then this will be ignored when the entry is saved. You can choose a mode from the pull-down menu, or enter a single letter for a custom mode if you desire. If you need information on Addressing modes, you may wish to refer to the section called “Label Addressing Mode Specification Command Line”.

The Address is the current address where you clicked. This can be edited. If a range is desired, you can specify the range in any format acceptable to OS9Disasm. The above link should give more details if needed. Again, as with "Rename Label", you can change this address to any other you wish.

Saving The Data Boundary. Clicking Save saves this command as an OS9Disasm Data Bounds command. Please refer to Saving the entry for details on how the command is saved and positioned in the buffer.

The Set Addressing Mode Window

This window looks like:

The Set Addressing Mode Window

Here we set an Addressing Mode specification for an address or address range. At the top center of the window, we see a single letter. This represents the label class for the instruction in the current line. It is obvious that this is not editable.

In the Addressing Mode box, we can select the mode in which we wish the operand to appear either by choosing from the drop-down box or by typing in our own choice. Label Classes are discussed in more detail in the section the section called “Label Addressing Mode Specification Command Line”.

The Address box works identical to that of Data Bounds described for the section called “The Define Bounds Window”

The Offset from Address. This allows you to specify an offset from the destination address. Please refer to the description of offset addressing below the section called “Label Addressing Mode Specification Command Line” for details on offset addressing.

The Offset entry utilizes a drop-down menu along with an entry box. You can type in an offset or click the arrow to select a previously-selected item. On startup, the list is empty, except for a single entry "NONE". If this is selected, or an empty line is there when you click "OK", then no offset is inserted in the command line.

As the editing session progresses, a history of offsets is maintained. Each time you enter a new offset, if it does not match exactly an entry already present, then it is added to the list. This is a convenience feature whereby you aren't required to enter the same offset repeatedly. Note that a functionally identical entry, but differing by a space will be added. There is no harm in this except that it adds a bit of "clutter" in repeated functionally identical entries.

Note

There is no provision to save these entries. All are lost upon close of the editing session.

Saving the entry. Clicking Save will insert the OS9Disasm command apprpriate for this entry in the Command file buffer on a separate line at the position of the cursor. If the cursor is at the begin of the line, it will insert the line prior to that line. In any other event, it wll insert it after that line. Clicking Cancel will close out the window and abort the command.

The Label File Popup

Right-clicking in the Label File window causes a popup window to appear there.

The Label File Popup

Search Labels

The Search Labels option enables a search through the Label File buffer. You can perform a search on the Label or the Address fields in this list. See the section called “Performing Searches” for details on how searching works.

Open Label File

This option allows you to load a different (or reload the same file) into the buffer. All old data in the buffer is cleared out and replaced with the new.

Save LBLFILE (as)

Save LBLFILE and Save LBLFILE AS.. saves the buffer data. If the data was loaded from a file, Save LBLFILE saves the buffer without prompting for a filename. If the data has all been entered and no lblfile had been loaded, Save LBLFILE acts as Save LBLFILE As.., where it will prompt yo for a filename before trying to save.

Insert line after

This will allow you to enter a line and insert it after the selected line.

Delete

The Delete option removes the currently selected line from the buffer.

Performing Searches

The Search Dialog

Searches can be made in the Listing and the Label buffers. They can be made on specific columns, and the search will apply to only that column, ignoring any matches that might occur in any other column.

Selecting a column to search.  You may specify which column in which you wish to search by clicking on the desired column in the Column to Search box. The columns shown will depend upon the buffer being searched.

Enter Search Text.  The text for which you wish to search can be specified in the text box labeled by that name. After a search has been made in a window, the last search will be remembered and be redisplayed on the next search. The memory is separated, that is, a search on the Listing buffer will bring up the last text searched in that buffer. A search in the Labels buffer will bring up the last search for that buffer.

Executing the search.  Clicking on the Search Backward or the Search Forward button will execute the search. Clicking on the Cancel button will abort the search.

If a row is selected, the search will begin at the next row according to the direction of the search. If the end of the buffer is reached, the search will jump to the opposite end of the buffer and continue until it comes back to the place where the search originally started. It will stop upon the first match. However, to continue the search after a match, simply begin the search at the row of the match, or any row past the match, in the direction of the search, and redo the search. The search text will already be set up for the current search text.

Embedding Label Classes in Files

OS9Disasm will have several Label Classes that are defined by default and are inserted on startup. However, you will find that, with a project of any appreciable size, that you will probably wish to have additional classes defined besides those already defined.

There is a convenient way to reload these definitions into G09Dis rather than having to reenter them each time you run the program. Once you have a label file started, you can embed these classes in a comment line[4]. All you do is begin a line with "*\ " followed by the letter designating the label class. What follows does not matter. The spacing is totally irrelevant, but a recommended format is 

*\<space> <class> <space> - <space> <description>

By using embedded commands, as the label file is loaded, the custom label classes are automatically added. If the label file is reloaded, they won't be reappended as long as the line remains exactly as it was. If one of these definitions is loaded and the string is exactly identical to one already present, it won't get reloaded. However, if you should, for example, load a file, edit the description for one of these defs, save and then reload the file, you will get duplicate entries for that class. It won't hurt anything except you'll just have two entries in the drop-down list.

Note

You can also write a separate file for these embedded label definitions and load them from the Tools->Edit Amode menu option.

G09Dis and Finding Files

If you're not careful, you may encounter problems with G09Dis that might need to be noted.

  • If you run G09Dis from the command line, you can specify a full path to the program to get it to execute. If it is not in the PATH environment, and not in the current directory, this may be necessary.

    If G09Dis is in the current directory .  If you are in MS Windows, often simply issuing the command G09Dis is sufficient. However, on any system, if "." is not in the PATH environment, then you will have to issue the command ./g09dis

    Note

    For Windows use, it may be difficult to operate G09Dis by clicking on the icon in Explorer unless this copy of G09Dis happens to be in the same directory as your label files, especially.

For greater details on locating files and possible ways to avoid difficulty, please refer to the section called “Install the binaries where the system can find them”.

Finding the label file specified in the command file

While the full path binary to be disassembled and the command file can be specified in the Options menu, the label file could cause some difficulty. OS9Disasm searches for the label file in the following manner:

  1. OS9Disasm first searches in the current directory. (This would be in this case the directory from which G09Dis was called from).
  2. Next the "default" directory is searched
  3. The path specified for the label file, if not absolute (preceded by a "/" (or "\" on MS-Win), relative to the absolute path to the command file is tried.
  4. Finally the HOME directory is searched.

If the file is not found in any of these places, an error message is issued. In the case of a label file, the program does not abort, but continues, but the labels defined in this file will not be included.

Visual Preferences

The visual appearance of the windows in the display can be varied according to taste. The font for the text in the windows can be changed, and the color of the text and the background, also can be altered. This can be done for any of the listings, command file, or the labels file windows.

The reason for desiring a different appearance can be simply for aesthetic reasons, or it might be to make it easier to read.

This can be done in two ways, either from a file in the home directory or at any time during the execution of the program.

The RC file

An rc file can be used to set up certain conditions each time the program is started up. It can be used to set fonts and colors for certain sections of the display. An in-depth explanation of this can be found in the GTK documentation. We will give the parts that pertain to us here.

An example file as it might pertain to G09Dis might be: 

            style "txtstyle"
            {
                font_name = "monospace 9"
                text[NORMAL] = "#df1010"
                base[NORMAL] = "#d0d0d0"
            }

            widget "*txtwin" style "txtstyle"

            style "lblstyle"
            {
                font_name = "Newspaper 12"
                text[NORMAL] = "blue"
                base[NORMAL] = "#c0c0c0"
            }
            widget "*.lblwin" style "lblstyle"

The word "style" is a keyword defining a set of parameters that will apply to an element, or group of elements, in the display. Following this is the name of the style which we are defining. It doesn't matter what name we use here, it will be used later in the file to determine what style we are applying to a group.

Following this style name is a list of settings to be applied to the style. These settings must be enclosed in a set of brackets ("{" and "}"). Any number of styles can be defined and in any order. The only requirement is that the style be defined before it is assigned to a group.

In the example above, we have defined a pair of styles. Below this, we apply the style "txtstyle" to a group. The keyword widget and style means that the style txtstyle should be applied to all widgets with the name txtwin. The asterisk (*) in this case is required. For our purpose, the period (.) is not necessary. When present, it means that this style be applied only elements with the name txtwin. If it were absent, the style would be applied to all elements with names that end with txtwin.

We also define a style lblstyle. This defines a second style, and we apply this style to all widgets with the name lblwin.

We have two widget names with which we can apply styles. The first is named txtwin, and this refers to Listing Window and the Commands Window. Any style applied to txtwin will be appied to both. Of course, either can be changed during program execution.

The style lblstyle is applied to the the Label window. This window's properties can also be altered during the running of the program (see the section called “Changing fonts and colors during program execution”).

You might not wish to preset any styles (at least the font) on the Label window, as the default Sans 10 is pretty With the other windows, however, especially the Listing Window one might wish to see more of the line at one time, so a smaller font might be desired. On the other hand, one might wish to see the text more clearly, and desire a larger font, Therefore, one can specify a default text size for these two windows.

Styles we can alter. We will not deal with an extensive list of style features, although several might apply. We will only discuss the font and color features. If more details are desired, the reader is encouraged to consult the GTK documentation. We wish to alter the following features:

font_name

This determines the font the window will use. The definition is a text string. The best way to find out which fonts and font names are applicable, simply run the program and click on File->Fonts and search through the various fonts. Selecting one will display some sample text at the bottom of the window.

text[NORMAL]

This selects the color for the text. The keyword [NORMAL] is necessary. This represents the state of the text, and there are four other states, mostly referring to the condition of the mouse when and whether it is over the text. There are four other states, but for us, NORMAL is our primary concern.

The color can be defined in several ways. In any case, the value must be enclosed in quotes, either single (') or double (""). It doesn't matter which, but they must match on both ends of the value.

  • A color Name.  - The color can be defined by simply a color name, such as "blue", "green", or the like.

  • Numeric definition for the color.  - The color can be defined by the use of a hexadecimal number representing the intensity of each of the three basic colors. These colors, in order, are 'RED', 'GREEN', and 'BLUE'.

    This number is preceded by a pound sign ("#"), and can be expressed as either a 24-bit number, such as '#df1010' (a red color), or as a 72-bit value, such as '#10001000df00' (rather blue).

base[NORMAL]

This defines the background color for the window. The syntax for the selection is the same as for the foreground.

It is not required that we define a style for any of the widgets. If either is not defined, the default will be black text over a white background, with a Sans 10 font, which is pretty much acceptable.

Changing fonts and colors during program execution

The font and colors for all three windows can be reset at any time during the execution of the program. This can be done by clicking the File->Fonts from the Main Menu bar.

When you do this, you will see the following window pop up:

The Font Selection Window

Selecting a window to affect. On the left side of the window, you see a set of radio buttons with the name of the window you wish to affect. The settings you choose will apply to only one window - the one you have selected here. By default, the Listings window will always be chosen upon entry. To affect another window, simply click the button beside the name of the desired window.

Performing the actual font/color selection. To do the actual font or color selection, click on the correct button in the section on the right. The top button selects the font, and the current font together with its point size for the currently-selected window is displayed inside this button. Clicking this button will bring up a Font Selection window. This window is very much self explanatory so no effort to describe it will be made. The color selection buttons are below the font selection button. Clicking the button labeled Background or Foreground will bring up a Color Selection window, which will allow you to select a color for the respective field. Note that the current color for the currently-selected window is displayed in the respective color-select button.

Note

Choosing a different Window on the left will cause the buttons on the right to display the font or colors for that window.

Aborting with no changes. After making a font or color selection, by clicking OK this change takes effect upon the window immediately. Clicking Cancel in the font or color selection window will abort that choice attemt. In either case, however, the main Font/Color choice window will remain up until OK or Cancel is pressed. Clicking OK will exit the Font choice window and all changes will remain. However, if Cancel is pressed, all changes made during that session will be undone , and all windows will revert back to the state they were in upon entry into the session.
