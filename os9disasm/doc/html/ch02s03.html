2.3. Building & Installing the Package

The packages now come configured with autotools. This means that the configure process should be able to detect under what system the process is running and set some compile parameters to suit your situation. This should make it easier for the programs to be compiled under differing systems.

2.3.1. Running the configure script

The first thing you need to do after cd'ing into the top level of the distributions directory is to issue the command ./configure. This will inspect the system under which it is running and create various files, and especially the Makefiles that will handle the actual building of the system.

Don't mess with the configuration files!

Unless you know what you're doing and see that something in the configure setup needs to be fixed, please heed the following.

There are a few files that you should never touch unless you have the autotools package. If the timestamp of any of these are not as they should be, a complete reconfigure may be required. If you do not have the autotools utilities, you cannot do this, and even if you do, if it's version is not at least that of the one which originally built the configure scripts, then a complete reconfigure might be required.

The files that should not be touched are as follows:

  • Makefile.am
  • Makefile.in
  • configure.ac
  • configure.in
  • aclocal.m4

If you do find that the system is trying to do a reconfigure and you are stuck, you can always remove your install of the package and reexpand it from the original tarfile.

2.3.1.1. Options passed to configure

You can review the options accepted by configure by issuing the command ./configure --help. Most are the standard options that all configure scripts accept, but there are a few specific to this build.

--with-g09dis

G09dis by default is not compiled with OS9Disasm. In order to compile g09dis, pkg-config must be installed on the system. Since some systems might not have this, the configure process does not include g09dis in the compilation. If this option is not included, the Makefile for g09dis will be made, but will not be correct, and the g09dis directory will not be processed when the make command is issued.

2.3.2. Building the program(s)

After you cd into the top level of the directory, issuing the command make will build the program OS9Disasm. This will build the program, but won't install it.

Note

You are not required to build from the top-level directory. The system is designed such that you can cd into any subdirectory and run make on just that directory. It will, of course, recurse into any subdirectories below that level, but won't bother anything in a sibling or higher directory outside that particular level.

This holds true for any of the make commands. The command will apply over only that directory and any directories below it. No higher directory or any outside this directory will be involved.

2.3.3. Installing the program(s)

To install the program, you can issue the command make install (at any directory level you choose). This can be done either after the program has been built, or before. If it hasn't been built, issuing this command will cause it to be built before attempting to do the install. By default, this will install the OS9Disasm binary in the directory /usr/local/bin/.

2.3.3.1. Installing in an alternate location

If you wish to install it elsewhere, then you can either simply copy the binary files to whatever destination you want after running the command make, or, if you wish to install the program(s) at a different location using make install, the you can include the option --prefix=<alternate-location> when you run configure.

2.3.3.2. Install the binaries where the system can find them

It may be very advisable to carefully contemplate the location where you will install the binaries. This is especially true for OS9Disasm if you are planning on calling it from within g09dis. If you disassemble a binary from the menu in g09dis, a shell computeroutput will be issued which will call up OS9Disasm to do the work. The call will be of the form 

"os9disasm program <etc>"

It would be a bit difficult, and harsh, to hardcode a specific location into the program whereby everyone would be required to install OS9Disasm at some specific location. There are just too many differences between the various systems and people have preferences as to their install locations. Therefore the call is called up in somthing of an open manner, and it's left up to the operating system to find OS9Disasm. The following gives some insight in how this can be assured.

Linux and Unix-type systems.  To find the directories searched in locating an executable program, from the shell prompt, issue the command 

echo $PATH
This will give a list of directories, separated by a colon, searched when looking for a command. They are searched in the order in which they appear.

Note

For security reasons, be aware that most Linux/Unix systems don't include the current working directory in the search path.

Windows Command prompts. If you're running from a POSIX shell, (msys, cygwin, or sh/bash/etc from an MS window, you can view the search PATH with the command as above, 

echo $PATH
or, if from the MS command, 
echo %PATH%
In msys/cygwin, you can add more search directories by editing one of the configure files. Please see the documentation for the system for details. To alter the windows PATH, you will need to edit the autoexec.bat file. To do this, you can either add your new directory to an existing PATH statement or add a line such as 
set PATH=%PATH%;your-new-directory
Note that on most systems in Windows, the current working directory is normally included in the search path. Therefore, you should be able to call OS9Disasm successfully if OS9Disasm is in the directory you are in when you call OS9Disasm. In other words, if the command 
ls os9disasm
or 
dir os9disasm
if you are at a DOS prompt succeeds, then you should be OK if you call g09dis as long as you don't cd somewhere else before doing so.

Note

You can alter the PATH variable at any time by by issuing the above command in an MS window, but this change will be lost when the window is shut down and won't be in effect when starting a new window, or won't be available to the global Windows system.

Running g09dis from the Explorer Window. I'm not entirely familiar with how this will work. If you do this, probably the safest route would be to make certain that OS9Disasm is in a directory listed in PATH.

2.3.4. The Documentation files

Due to all the differences in systems, it has been decided at this time to not do an install of the documentation with a make install command from a higher directory. In fact, at this time, nothing is done with the documentation by makes from a higher level. The html files are located in the doc/html directory. These can be manually copied to any location you desire. In addition, the xml source is available in the doc directory, and fresh html documentation can be created by issuing the command make in the doc directory itself.

Building your own documentation. Since the source is available in the doc directory, you have the option of building the documentation in any format you desire. Just run any docbook compiler on os9disasm.xml and create any output you wish. The file os9disasm.xml will include all the rest of the files.

2.3.5. Installing the Standard Label Files

In the top level directory, there is a directory named defs. In this directory, you will find several files ending in "lbl". These are the standard label files. Copy all these files into $HOME/coco/defs/. This should work well for a single user system, or on any system where only one person is using this program.
