4.4. The System Name File

The system name file is used by OS9Disasm to supply the proper names to the function bytes used in OS9 system calls, via the OS9 opcode. For instance, the call to end a program, OS9 F$Exit, is actually SWI2 followed by the byte value $06. To properly disassemble this code, OS9Disasm must know that, after the SWI2, $06 means F$Exit.

The system name file is a text file having the same format as a label file. All the labels in the file should be set up with the system name label class '!'. Actually, OS9Disasm ignores the label class when processing the system name file, but it should still be there, so that the file can be used as a normal label file.

The system name file defaults to the file Sysnames for OS/9 or Motorola format object code. For CoCo DOS or FLEX format, no system name file is used, and no OS9 opcodes will be produced. To allow OS9 opcodes in a CoCo DOS or Flex format object files, you must specify the option -f=Sysnames yourself, either on the OS/9 command line or in a command line in the comman file.

If you wish to add or change a system function name, you have several options. First, if the change is to be made only in a single program, it is best to include an entry in your own label file, with label class '!'. Second, if the changes are more extensive or more permanent, you may edit the default file, or replace it with one of your own with the -f option.

The file miscos9lbl. 

When disassembling OS9 files, there's one other "standard" file that will usually be needed. It's not read by default, you will need to specify it either on the command line or in the command file. Please refer to it for its usage. It defines several commonly-used classes, some of which are error codes (class E), set/get stat codes (class S), and several others. The name is miscosolbl.

4.4.1. Overriding Default Label Definitions

If you include these default files, but wish to give any of these values a different name, you can simply redefine them in a later label definition. The default label files are scanned first, then the user-specified label files are scanned in the order in which they are specified, beginning with the files named on the command line. No check is made when inserting a label def as to whether it's already been defined or not - well, actually, OS9Disasm has already defined this label with its own definition (usually either class L, D, or X before the label files are read. Therefore, if a label is defined, and later redefined, then this later definition is applied.

This will work if you create your own custom label files. You can override any of the "standard" label names in your custom file or override your own custom definitions in subsequent files.
